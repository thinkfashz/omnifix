1 Setear cookie tenant_status en el login
Crítico

El middleware lee la cookie tenant_status para redirigir a /admin/plan-suspendido si el tenant está suspendido o cancelado. Sin embargo, esta cookie nunca se escribe. El resultado: un tenant suspendido puede entrar al panel sin restricciones.

Archivo a modificar

src/app/api/admin/login/route.ts

Cambio a aplicar

Después de construir la respuesta de login exitoso, agregar el header Set-Cookie para tenant_status:

// Después de crear la respuesta con el JWT de sesión...
const response = NextResponse.json({ ok: true });

// Setear cookie de sesión JWT (ya existente)
response.cookies.set('admin_session', token, {
  httpOnly: true,
  secure: process.env.NODE_ENV === 'production',
  maxAge: 60 * 60 * 24 * 7,
  path: '/',
});

// ★ NUEVO: setear tenant_status (NO httpOnly para que Edge lo pueda leer)
response.cookies.set('tenant_status', tenantStatus, {
  httpOnly: false,   // Edge runtime puede leer cookies no-httpOnly
  secure: process.env.NODE_ENV === 'production',
  maxAge: 60 * 60 * 24 * 7,
  path: '/',
  sameSite: 'lax',
});

return response;

Donde tenantStatus viene del query del tenant al hacer login. Consultar el tenant después de validar credenciales:

// Obtener status del tenant para la cookie
const { data: tenant } = await insforge.database
  .from('tenants')
  .select('status')
  .eq('id', tenantId)
  .single();

const tenantStatus = tenant?.status ?? 'active';
Nota: La cookie debe ser httpOnly: false porque el Edge runtime (middleware) lee cookies a través de request.cookies.get() y no puede acceder a cookies httpOnly desde el cliente. Edge sí puede leer headers de cookies del request entrante.
2 Crear ruta /api/platform/notify-suspension
Crítico

El cron expire-trials llama a esta ruta con fetch() para notificar suspensiones, pero la ruta no existe. Esto genera un error 404 silencioso en producción y el tenant no recibe el email de aviso.

Archivo a crear

src/app/api/platform/notify-suspension/route.ts

Implementación completa

import { NextRequest, NextResponse } from 'next/server';
import { insforge } from '@/lib/insforge';
import { Resend } from 'resend';

export const dynamic = 'force-dynamic';
export const runtime = 'nodejs';

export async function POST(request: NextRequest) {
  const secret = request.headers.get('x-platform-secret');
  if (secret !== process.env.PLATFORM_ADMIN_SECRET) {
    return NextResponse.json({ error: 'No autorizado.' }, { status: 401 });
  }

  const { tenant_id } = await request.json();
  if (!tenant_id) return NextResponse.json({ error: 'tenant_id requerido' }, { status: 400 });

  const { data: tenant } = await insforge.database
    .from('tenants')
    .select('name, owner_email, slug')
    .eq('id', tenant_id)
    .single();

  if (!tenant) return NextResponse.json({ error: 'Tenant no encontrado' }, { status: 404 });

  const resend = new Resend(process.env.RESEND_API_KEY);
  const adminUrl = `https://${tenant.slug}.fabrick.cl/admin`;

  await resend.emails.send({
    from: process.env.EMAIL_FROM ?? 'Fabrick <noreply@fabrick.cl>',
    to: tenant.owner_email,
    subject: 'Tu cuenta Fabrick ha sido suspendida',
    html: suspensionEmailHtml(tenant.name, adminUrl),
  });

  return NextResponse.json({ ok: true });
}

function suspensionEmailHtml(name: string, adminUrl: string): string {
  return `<!DOCTYPE html>
  <html><body style="font-family:sans-serif;background:#0a0a0a;color:#fff;padding:40px">
    <h2 style="color:#f59e0b">⚠ Cuenta suspendida — ${name}</h2>
    <p style="color:#aaa">Tu período de prueba ha vencido y no registramos un pago activo.</p>
    <p style="color:#aaa">Tus datos se conservan por 30 días.</p>
    <a href="${adminUrl}" style="display:inline-block;background:#10b981;color:#000;
       font-weight:700;padding:12px 24px;border-radius:8px;text-decoration:none;margin-top:16px">
      Renovar suscripción
    </a>
  </body></html>`;
}
3 Constraint UNIQUE (clave, tenant_id) en tabla configuracion
Crítico

El upsert de settings usa onConflict: 'clave,tenant_id', pero la tabla configuracion solo tiene UNIQUE(clave). Si se ejecuta un upsert multi-tenant, falla en producción con error de conflicto.

SQL a agregar en la migración

-- Eliminar constraint antiguo (solo clave)
ALTER TABLE configuracion
  DROP CONSTRAINT IF EXISTS configuracion_clave_key;

-- Crear constraint compuesto
ALTER TABLE configuracion
  ADD CONSTRAINT configuracion_clave_tenant_unique
  UNIQUE (clave, tenant_id);

-- Índice para búsquedas por tenant
CREATE INDEX IF NOT EXISTS idx_configuracion_tenant
  ON configuracion (tenant_id);
¡Atención! Antes de ejecutar, verificar que no existan filas duplicadas con la misma clave y diferente tenant_id. Si existen, limpiar primero: DELETE FROM configuracion WHERE tenant_id != '00000000-0000-0000-0000-000000000001' AND clave IN (SELECT clave FROM configuracion GROUP BY clave HAVING COUNT(*) > 1);

4Tenant-scope en rutas de listado de admin
Importante

Las rutas de listado (GET /api/admin/orders y GET /api/admin/productos) no filtran por tenant_id. Un admin de Tenant A podría ver los pedidos/productos de Tenant B. Solo el endpoint PATCH /api/admin/orders/[id] fue tenant-scoped.

Archivos a modificar

src/app/api/admin/orders/route.ts src/app/api/admin/productos/route.ts

Patrón a aplicar en ambas rutas

import { getAdminTenantId } from '@/lib/adminApi';

export async function GET(request: NextRequest) {
  const tenantId = await getAdminTenantId(request);

  const { data, error } = await insforge.database
    .from('orders')   // o 'products'
    .select('*')
    .eq('tenant_id', tenantId)  // ★ agregar esta línea
    .order('created_at', { ascending: false });

  if (error) return NextResponse.json({ error: error.message }, { status: 500 });
  return NextResponse.json(data);
}

6Flujo de cambio de contraseña temporal
Importante

El sistema de provisioning genera una contraseña temporal que se envía por email. No hay ningún mecanismo que obligue al usuario a cambiarla en el primer acceso.

Cambios requeridos

  4. Crear página src/app/admin/cambiar-password/page.tsx con form de nueva contraseña

3Infraestructura para go-live Antes de abrir al público

7DNS wildcard *.fabrick.cl
Infraestructura

El routing por subdominios requiere que cualquier *.fabrick.cl apunte a Vercel. Sin esto, los subdomains de tenants no resuelven.

Pasos en el panel DNS (Cloudflare u otro)

  1. Crear registro CNAME: * → cname.vercel-dns.com (Proxy OFF si usas Cloudflare)
  2. En Vercel Dashboard → Settings → Domains → Add domain: *.fabrick.cl
  3. Vercel genera un challenge TXT; agregar en DNS panel
  4. Verificar con: curl -I https://test.fabrick.cl (debe devolver 200 o redirect de Next)
Cloudflare: Si usas Cloudflare como proxy, el wildcard CNAME debe tener el proxy desactivado (nube gris). De lo contrario Vercel no puede renovar el certificado SSL wildcard.
8Ejecutar migración SQL en producción
Infraestructura

El archivo scripts/add-multitenancy.sql contiene todas las tablas, columnas, índices y vistas nuevas. Sin ejecutarlo, el sistema falla en runtime.

Orden de ejecución

  1. Hacer backup completo de la DB (snapshot en InsForge u otro)
  2. Ejecutar en una DB de staging primero y verificar que las consultas existentes no se rompan
  3. Ejecutar en producción en ventana de mantenimiento (baja carga)
  4. Verificar con: SELECT COUNT(*) FROM tenants; — debe devolver al menos 1 (el tenant default)
  5. Ejecutar el SQL del constraint de configuracion (Tarea #3 de esta guía)

Verificación post-migración

9Registrar webhook en MercadoPago
Infraestructura

Sin el webhook registrado, MercadoPago no puede notificar cuando un cliente paga. La activación automática de cuentas no funcionará.

Configuración en el panel MP

  1. Ir a: MercadoPago Developers → Tu aplicación → Webhooks
  2. URL del webhook: https://fabrick.cl/api/platform/subscriptions/webhook
  3. Eventos a suscribir: subscription_preapproval y subscription_preapproval_plan
  4. Copiar el secret generado por MP → pegar como PLATFORM_MP_WEBHOOK_SECRET en Vercel
  5. Hacer un pago de prueba y verificar que el endpoint devuelva 200 en los logs de MP

5Variables de entorno completas

Todas deben configurarse en Vercel → Settings → Environment Variables con scope Production.

PLATFORM_ADMIN_SECRET Secret aleatorio (64 chars). Protege /platform y /api/platform/*. Generar con: openssl rand -hex 32
PLATFORM_MP_WEBHOOK_SECRET Secret que entrega MercadoPago al crear el webhook. Se usa para verificar firma HMAC-SHA256.
MP_CLIENT_ID Client ID de la app MercadoPago (panel de developers.mercadopago.com)
MP_CLIENT_SECRET Client Secret de la app MercadoPago. Nunca exponerlo en el cliente.
RESEND_API_KEY API key de Resend.com para envío de emails transaccionales. Crear en resend.com/api-keys
EMAIL_FROM Dirección de origen de emails. Ejemplo: "Fabrick <noreply@fabrick.cl>". Debe estar verificada en Resend.
CRON_SECRET Bearer token que Vercel Cron envía en Authorization header. Generar con: openssl rand -hex 24
NEXT_PUBLIC_APP_URL URL base pública de la plataforma. Ejemplo: https://fabrick.cl
INSFORGE_URL URL del proyecto InsForge (BaaS). Ya configurada, verificar que sea producción.
INSFORGE_ANON_KEY Clave anon pública de InsForge. Ya configurada.
INSFORGE_SERVICE_ROLE_KEY Clave de servicio de InsForge (acceso sin restricciones). Solo en servidor, nunca en cliente.
ADMIN_SESSION_SECRET Secret HMAC-SHA256 para firmar JWTs de sesión admin. Generar con: openssl rand -hex 32
